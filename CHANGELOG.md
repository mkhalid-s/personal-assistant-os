- Approval integrity binding: when an action is approved via `approve_and_execute` the current payload's canonical SHA-256 is pinned to the row alongside the approval timestamp. Before execution MYOS refuses any action whose payload no longer matches the pinned hash (`payload_hash_mismatch`) or whose approval is older than `MYOS_APPROVAL_TTL_SECONDS` (default 24h — `approval_ttl_exceeded`). Both refusals record a failed execution receipt and an `approval_integrity_block` event so tampering and stale approvals are auditable rather than silent.
- Execution receipts now include an `approval_integrity` context (schema `myos.approval_integrity.v1`) with `payload_hash_verified`, `approved_age_seconds`, `approval_ttl_seconds`, and `ttl_remaining_seconds`; `myos execution-receipt list/show` surfaces those signals beside the verification receipt so operators can see integrity status at a glance.
- Zero executor artifacts and action metadata now surface `timed_out`, `timeout_seconds`, `stderr_bytes`, and `stderr_truncated` so operators can see wall-clock termination and stderr volume without inspecting raw executor output. The disposable worktree is still cleaned up on every code path (including timeouts), and a regression test simulates a hanging Zero process to prove both the timeout guard and worktree cleanup.
- Zero factory runs now honor `MYOS_ZERO_TIMEOUT_SECONDS` as a global wall-clock cap when the factory context does not specify its own `timeout`, mirroring the approval-TTL env-override pattern. `myos doctor` reports the effective Zero timeout and whether it came from the env or the default.
- Populated the previously-empty `tests/test_backlog.py` with an executable CLI inventory smoke test: every registered command in `command_registry.COMMAND_SPECS` must be reachable in `build_parser`, must render `--help` cleanly under argparse, and must eventually route to a `func` handler through any depth of nested required subparsers. Runs in-process in ~0.1s and complements the metadata-focused command contract audit already in `release-check --strict`.

- CI static-checks job runs `ruff check` fatally on the safety-critical modules (`command_registry`, `privacy`, `execution`, `db`, `zero_executor`) and `mypy --strict` fatally on `command_registry.py` so the CLI command contract cannot regress type safety. `ruff check` + `ruff format --check` on the whole tree, `mypy --strict` on the remaining safety-critical modules, and `pip-audit` all run as advisory checks (continue-on-error), giving contributors visibility while a follow-up slice progressively flips each check to fatal. Both suppress the normal text output when `--json` is set so downstream tooling can parse them with `python -m json.tool` or `jq`. Command contract examples for both include the `--json` variant so the registry stays the source of truth.
